How Elastic Works
- Elastic creates an inverted index. So for example if there is a document then it will store which all words are there in that document and then it also maintains the other documents where that word is. So when it has to do a text search it knows which all documents have that word.
- There are other methods also like stemming, tokenization that help with the fuzzy text search also.
- These indexes are divided into shards. 
- A shard is a lucene index.
- and these shards are further divided into segments.
- Segments are immutable, that is one a document is  indexed it is not changed, change operations are supported by actually deleting a record and adding a new one.
- To make sure these deleted records are removed these segments are merged into larger segments. This also improves the search efficiency as we will not have to look for the data in a lot of segments. This is called the merge process.
- Shards are generally replicated for higher availability

Optimisations
- Using auto generated ids, this is useful becasu ES will not have to check if the id is already there in the db which is an expensive process.
- ensure that that each shard has enough memory buffer size.
- Use bulk requests till the time the response starts to plateau, but not to increase it over couple tens of mbs per req
- tune the refresh interval
- disable replicas
- disabled swapping
- use autogenerated ids
- 20 shards per GB of RAM
- Batch replication vs real time replication
- [`indices.memory.index_buffer_size`](https://www.elastic.co/guide/en/elasticsearch/reference/current/indexing-buffer.html "Indexing buffer settings") is large enough to give at most 512 MB indexing buffer per shard doing heavy indexing
	- The default is `10%` which is often plenty: for example, if you give the JVM 10GB of memory, it will give 1GB to the index buffer, which is enough to host two shards that are heavily indexing.
- [Hot Spotting](https://www.elastic.co/guide/en/elasticsearch/reference/current/hotspotting.html "Hot spotting") can occur when node resources, shards, or requests are not evenly distributed.
- give memory for file system cache
- [take control of your mappings](https://www.elastic.co/guide/en/elasticsearch/reference/current/explicit-mapping.html "Explicit mapping").
- Aim for shards of up to 200M documents, or with sizes between 10GB and 50GB

- Searching Optimizations
	- Searching runs on a single thread per shard
	- index shrinking
	- In most cases, a small set of large shards uses fewer resources than many small shards.
	-  You can use the [Cluster stats API](https://www.elastic.co/guide/en/elasticsearch/reference/current/cluster-stats.html "Cluster stats API") to get the heap overhead of the total size of all mappings after deduplication and compression.
